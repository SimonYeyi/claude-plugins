# bug-book 需求规格文档

> AI 错题集：让 AI 记住历史问题的现象、根因、解决方案，防止重复犯错。

## 背景与目标

在开发过程中，AI 经常在修改新功能时破坏之前已实现的功能，导致开发者需要反复说明问题、重新解决、重新测试。bug-book 插件通过自动记录问题信息、主动召回历史 bug、持续整理错题集，让 AI 能够记住过往的错误和解决方案，从根本上减少重复犯错。

## 核心功能

### 1. 记录

AI 在每次被用户纠正或发现问题时，将问题的完整信息记录到本地数据库。

- **小问题自动记录**：预估分 < 20，影响范围小，修复简单
- **复杂/二次问题直接记录**：预估分 >= 20，后台直接记录（`verified=False`），不等用户确认
- **修复审核**：用户确认修复时，展示记录供用户审核，用户确认后再更新 `verified=True`
- **懒初始化**：首次记录时自动创建数据库，用户无感知
- **用户主动记录**：用户可以主动触发记录，AI 主动询问核心字段后写入（用于记录不在当前上下文中的问题）

### 2. 召回（预防）

AI 在改代码前主动查询相关 bug，主动预防而非事后补救。

- **会话开始时批量召回**：基于当前项目文件，召回相关历史 bug
- **改代码前按需检查**：Edit/Write 前查询当前文件的 bug 记录
- **主动提醒**：搜索到相关 bug 时主动提示用户
- **未验证 bug 优先处理**：未验证说明可能还没修好，AI 更加谨慎

### 3. 整理

定期维护错题集的健康度，避免无效数据堆积。

- **路径有效性检查**：检查 `autoRecall` 路径是否还存在代码中，不存在则提示用户标记失效
- **归类相似问题**：按根因/标签归类，提示合并重复记录
- **长期未验证提醒**：超过 30 天未验证的记录，提示用户确认
- **按重要性排序**：分数高的 bug 排在前面，保证 AI 优先记住最重要的

### 4. 搜索

快速定位相关 bug，支持多种查询方式。

- **关键词搜索**：在标题/现象/根因/方案/标签/关键词中全文搜索
- **路径召回**：基于文件路径匹配相关 bug
- **按模式召回**：基于 `autoRecall` 匹配模式精准召回
- **查看详情**：查看某条 bug 的完整信息

## 数据模型

### bugs 表（主表）

| 字段 | 类型 | 说明 |
|------|------|------|
| id | INTEGER | 主键 |
| created_at | DATETIME | 创建时间 |
| updated_at | DATETIME | 更新时间 |
| title | TEXT | 标题（≤20字） |
| phenomenon | TEXT | 现象描述 |
| root_cause | TEXT | 根因分析 |
| solution | TEXT | 解决方案 |
| test_case | TEXT | 测试用例 |
| status | TEXT | 状态：active / resolved / invalid |
| verified | INTEGER | 是否已验证（0/1） |
| verified_at | DATETIME | 验证时间 |
| verified_by | TEXT | 验证者：Claude / User |
| score | REAL | 综合分数 |

### bug_scores 表（评分维度）

| 维度 | 权重 | 分值范围 | 说明 |
|------|------|----------|------|
| importance | 2.0 | 0-10 | 功能重要性，核心功能 8-10 |
| complexity | 1.5 | 0-10 | 逻辑复杂度，多模块/异步 7-10 |
| scope | 1.0 | 0-10 | 影响范围，影响整个系统 8-10 |
| difficulty | 1.0 | 0-10 | 修复难度，需大重构 8-10 |
| occurrences | 1.0 | 累计 | 每次重复出现 +1 |
| emotion | 1.5 | 0-5 | 用户情绪，被纠正 +2，发火 +5 |
| prevention | 2.0 | 0-10 | 预防价值，高频场景 8-10 |

**总分计算**：各维度加权求和，权重可通过环境变量 `BUG_BOOK_WEIGHTS` 覆盖。

```
总分 = importance×2.0 + complexity×1.5 + scope×1.0 + difficulty×1.0
     + occurrences×1.0 + emotion×1.5 + prevention×2.0
```

| 阈值常量 | 值 | 说明 |
|---------|-----|------|
| `THRESHOLD_AUTO_VERIFY` | 30 | 预估分 < 此值自动验证，>= 此值需审核 |
| `THRESHOLD_HIGH_SCORE` | 30 | 高分阈值，需明确确认 |
| `THRESHOLD_OLD_BUGS_DAYS` | 30 | 未验证提醒天数 |

### bug_paths 表（相关路径）

| 字段 | 说明 |
|------|------|
| path | 相关文件路径 |
| is_old | 是否为旧路径（重构后保留） |

### bug_tags / bug_keywords / bug_recalls 表

用于搜索和召回的标签、关键词、匹配模式。

### autoRecall 机制

每条 bug 记录包含 `autoRecall` 匹配模式，当 AI 修改相关文件时自动召回。

- 使用文件路径前缀匹配（如 `auth/*`），避免使用太泛的关键词
- 重构后保留 `oldPaths`，支持旧路径匹配
- 整理时检查路径有效性，无效时提示用户更新

## 组件设计

### Skills

| Skill | 触发条件 | 职责 |
|-------|----------|------|
| bug-record | 被纠正/发现问题时 | 判断是否需要记录，收集信息，评分，写入数据库 |
| bug-search | 用户搜索/改代码前 | 关键词/路径/模式查询，展示结果 |
| bug-organize | 用户要求整理 | 路径检查，归类，去重，清理，生成报告 |

### Agent

| Agent | 触发时机 | 职责 |
|-------|----------|------|
| bug-curator | 会话开始/代码改动前 | 主动召回，分数累加，验证追踪，路径维护，已验证 bug 复发处理 |

### Scripts

| Script | 职责 |
|--------|------|
| init_db.py | 初始化 SQLite 数据库表结构 |
| bug_ops.py | 所有数据库操作的封装（CRUD、搜索、召回、整理） |

## 验证状态管理

- **自动记录的小问题**：写入时 `verified=True`
- **需审核的复杂问题**：写入时 `verified=False`，修复后更新为 `True`
- **已验证 bug 复发**：打回 `verified=False`，累加出现次数，追加备注
- **长期未验证（>30天）**：整理时提醒用户确认
- **验证时**：记录 `verified_at` 和 `verified_by`

## 存储策略

- **多项目支持**：每个项目有独立的 `bug-book.db`，互不干扰
- **懒初始化**：首次记录时自动创建数据库
- **按需查询**：AI 只查需要的字段，不加载全量数据

## 不在当前范围内的设计

- 自动降分：随着新问题分数累积，自然排序，不需要额外降分机制
- recurrence_count 冗余字段：当前通过 scores 表查询即可，不需额外字段
- 项目路径字段：当前插件安装在项目目录下，自然隔离
