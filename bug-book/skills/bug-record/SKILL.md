# Bug Record Skill

当发现或被指出问题时，将问题的完整信息记录到 bug-book 数据库中。

## Python 脚本执行规则

**【强制要求】禁止 cd 到插件目录！必须在当前项目目录执行 Python 代码。**

- ✅ 正确：保持在项目目录（如 `/home/user/my-project`）执行
- ❌ 错误：cd 到插件目录（如 `~/.claude/plugins/bug-book/scripts`）

如果需要使用插件脚本，使用绝对路径导入：
```python
import sys
sys.path.insert(0, "{CLAUDE_PLUGIN_DIR}/bug-book")
from scripts.bug_ops import add_bug, add_impact
```

## 评分标准

### 评分维度

| 维度 | 评分依据                                                        |
|------|-------------------------------------------------------------|
| 功能重要性 | 核心功能 8-10，辅助功能 3-7，周边功能 0-2                                 |
| 逻辑复杂度 | 涉及多模块/异步/并发 7-10，单点逻辑 3-6，简单逻辑 0-2                          |
| 影响范围 | 影响整个系统 8-10，影响模块 4-7，影响单个文件 0-3                             |
| **修复难度** | **需要大重构 8-10，需要改多个文件 5-7，单文件修复 0-4；用户多次验证不通过（AI 反复修复）每次+3** |
| 出现次数 | 首次出现 +0，重复出现 +1（可累加）                                        |
| **用户情绪** | **被用户明确纠正 +2（每次纠正都加），用户发火 +5；AI 自己调试失败不算**                  |
| **预防价值** | **高频场景 +8-10，偶发场景 +3-7，低频场景 +0-2；修改影响其他功能（回归风险）每个+5**       |

### 打分示例

| 场景 | importance | complexity | scope | difficulty | emotion | prevention | 总分 | 处理 |
|------|-----------|------------|-------|------------|---------|------------|------|------|
| 笔误：变量名拼错 | 2 | 1 | 1 | 1 | 0 | 1 | **9.5** | 静默记录 |
| 单文件单处bug：按钮样式错位 | 3 | 2 | 2 | 2 | 1 | 3 | **20.0** | 静默记录 |
| 单文件多处bug：同一函数逻辑错误，调用链复杂，需改5处 | 6 | 5 | 3 | 5 | 2 | 5 | **41.0** | 审核记录 |
| 多文件bug：session存储未持久化 | 7 | 5 | 4 | 3 | 2 | 6 | **42.5** | 审核记录 |
| 回归bug：用户第三次抱怨同样问题 | 8 | 6 | 6 | 4 | 3 | 8 | **63.5** | 审核记录 |
| 核心模块内存泄漏 | 9 | 8 | 8 | 7 | 0 | 7 | **66.5** | 审核记录 |

## 记录字段说明

### 字段说明

#### 业务字段（直接存储在 bugs 表）

| 字段 | 说明 | 示例 |
|------|------|------|
| **标题** | 简述问题（控制在 20 字以内） | “刷新页面 session 丢失” |
| **现象** | 具体表现 | “登录后刷新页面需要重新登录” |
| **根因** | 为什么会这样（如未知可填预估值，后续验证时更新） | “session 存储时未设置持久化” |
| **解决方案** | 怎么修复的（如未知可填预估值，后续验证时更新） | “添加 cookie 的 maxAge 配置” |
| **测试用例** | 如何验证修复（如未知可填预估值，后续验证时更新） | “登录后刷新页面，验证 session 保持” |
| **相关文件/路径** | 哪些文件相关 | `["src/auth/session.ts"]` |
| **标签** | 分类标签 | `["auth", "session"]` |
| **关键词** | 便于搜索，**必须包含中英文** | `["session", "会话", "cookie", "持久化"]` |
| **autoRecall** | 触发召回的模式 | `["auth/*"]` |

#### 评分字段（独立存储在 bug_scores 表）

**重要**：评分不是单个字段，而是 7 个维度的分值，通过 `scores` 字典传入：

```python
scores = {
    "importance": 7,      # 重要性
    "complexity": 5,      # 复杂度
    "scope": 4,           # 影响范围
    "difficulty": 3,      # 修复难度
    "occurrences": 0,     # 出现次数
    "emotion": 2,         # 用户情绪
    "prevention": 6,      # 预防价值
}
```

- ✅ **必须填写所有 7 个维度**，参考「评分标准」章节给出分值
- ✅ 即使不确定也要给出**预估值**，后续验证时可调整
- ✅ **总分自动计算**：API 会根据权重自动计算总分，无需 AI 手动计算

**重要原则**：
- ✅ **尽量填写所有字段**，不要留空
- ✅ **评分必须填写**：即使不确定也要根据「评分标准」给出预估值
- ✅ 如果某些字段无法确定（如根因、解决方案），可以填**预估值**或**推断值**
- ✅ 但必须在后续验证时**更新为准确值**，绝不保留预估值

### 数据一致性保证

**原子性操作**：`add_bug()` 函数使用数据库事务保证原子性：
- ✅ 主表（bugs）和维度表（bug_scores）的插入在同一个事务中
- ✅ 如果任何一步失败，整个事务会自动回滚，不会出现“只有主表没有评分”的情况
- ✅ AI 无需担心数据不一致问题，直接调用 API 即可

### 等义关键词提取规范

**为什么需要等义关键词？**
- 用户描述的是：“登录页面”
- Bug 记录可能另一种表示：“会话管理”
- 简单的 `LIKE` 搜索无法匹配语义相似的词

**关键词提取的两个核心来源：**

1. **从上下文中提取**
   - 用户描述中的关键词
   - 对话中提到的技术术语
   - 问题现象中的核心词汇

2. **从召回路径中提取**
   - 文件路径中的业务词和技术词
   - **重要**：需要从英文推断相关的中文同义词
   - 示例映射规则：
     ```
     form → "表单" + "新增", "添加", "编辑", "修改"
     list → "列表" + "查询", "筛选", "排序"
     auth → "认证" + "登录", "权限", "会话"
     user → "用户" + "注册", "资料", "账户"
     ```

**实际示例：**

```python
# 场景 1：用户说"用户表单添加有问题"，文件是 user_form.vue
# 来源 1 - 上下文："用户", "表单", "添加"
# 来源 2 - 路径推断：user → "用户", form → "表单" + "新增/添加/编辑/修改"

keywords = [
    # 从上下文提取
    "用户", "表单", "添加",
    # 从路径推断（英文 + 中文同义词）
    "user", "form", "新增", "编辑", "修改",
]

# 场景 2：用户说"认证会话超时"，文件是 auth_session.ts
# 来源 1 - 上下文："认证", "会话", "超时"
# 来源 2 - 路径推断：auth → "认证" + "登录/权限", session → "会话"

keywords = [
    # 从上下文提取
    "认证", "会话", "超时",
    # 从路径推断
    "auth", "session", "登录", "权限", "timeout",
]
```

## 自动记录机制

### 自动触发的强制信号（无需用户提醒）

当检测到以下任一情况时，**必须立即自动调用本技能**：
- ✅ 用户说“不对”、“错了”、“不是这样”、“回退”等否定词 + 指出具体问题
- ✅ 用户抱怨“之前不是这样的”、“又出现同样的问题”
- ✅ 用户的代码被手动修改或覆盖
- ✅ AI 发现自己引入了回归错误（新功能破坏了旧功能）
- ✅ **用户明确指出“修改 A 后 B 出问题了”**（需要记录影响关系）
- ✅ 排查过程中发现了根因但未提前记录

### 预估分决策逻辑

预估分 = Σ(各维度分值 × 权重)，权重：importance×2.0, complexity×1.5, scope×1.0, difficulty×1.0, occurrences×1.0, emotion×1.5, prevention×2.0

**静默记录**（预估分 < 30）：用户指出明显错误，影响范围小，修复简单。写入时 `verified=True`。

**记录后简单告知用户**：
```
✅ 已静默记录为 Bug #N（低优先级问题，已标记为已验证）
```

**审核记录**（预估分 >= 30）：
写入时 `verified=False`，修复时再审核。

**记录后告知用户（提供核心信息）**：
```
已记录为 Bug #N（审核记录），当前未验证。

**标题**：[标题]
**现象**：[现象]
**根因**：[根因，如未知则写“待补充”]
**解决方案**：[方案，如未知则写“待补充”]

当功能修复后请告知我，我来验证并更新准确信息。
```

### 决策流程

```
开始评估问题
    │
    ▼
打预估分（参考上表各维度）
    │
    ├── 预估分 < 30 ──→ 静默记录（verified=True）──→ 结束
    │
    └── 预估分 >= 30 ──→ 审核记录（verified=False）──→ 结束
```

## 主动记录模式（用户不在对话上下文中）

当用户主动说“帮我记录一个问题”（这个问题 AI 没有上下文），AI 应该先询问现象，然后立即生成标题，再根据情况引导：

### 第一步：询问问题信息

```
请描述一下遇到的问题：
- **现象**（必填）：具体表现是什么？
- **根因和解决方案**（选填）：如果已知，可以一并提供
```

### 第二步：生成标题

用户回答后，**AI 立即根据现象生成标题**（控制在 20 字以内）。

**标题原则：**
- 只从现象中概括，不涉及根因
- 简洁明了，便于快速浏览
- 例如：
  - 现象“刷新页面后需要重新登录” → 标题“刷新页面 session 丢失”
  - 现象“按钮点击没有反应” → 标题“按钮点击无响应”
  - 现象“数据库连接池在高并发时耗尽” → 标题“高并发时连接池耗尽”

**注意：** 标题不影响召回，因为召回会搜索现象、根因、关键词等多个字段。

### 第三步：确认问题状态

无论用户提供的是仅现象还是现象+根因+解决方案，都需要先确认问题状态：

```
我理解的问题是：**[生成的标题]**

这个问题是否已经修复？

- 如果已修复：我会记录并标记为已验证
- 如果未修复：是否需要我帮你解决这个问题？
```

**根据用户回答引导不同流程：**

#### 分支 1：用户说“已修复”

- **如果用户之前只提供了现象** → 需要补充根因和解决方案：
  ```
  好的，请提供：
  1. **根因**：为什么会这样？
  2. **解决方案**：怎么修的？
  ```
  
- **如果用户之前已提供根因和解决方案** → 直接记录：
  - AI 自动推断其他字段（测试用例、路径、标签、评分等）
  - 参考「记录字段说明」准备数据
  - 参考「API 调用指南」调用 `add_bug()`（verified=True）

#### 分支 2：用户说“未修复，需要你解决”

1. **立即开始排查**：分析问题根因
2. **执行修复**：修改代码解决问题
3. **修复完成后，记录问题**：
   - 此时 AI 已修复，但**需要用户验证**是否真的解决了
   - 记录为 `verified=False`（待验证状态）
4. **告知用户已修复并根据用户反馈修改验证状态**：
   ```
   ✅ 问题已修复并记录为 Bug #N
   
   **标题**：[标题]
   **现象**：[现象]
   **根因**：[根因]
   **解决方案**：[方案]
   
   请验证一下是否真的解决了？如果确认无误，我会将其标记为已验证。
   ```
5. **等待用户确认**：
   - 用户确认 → 调用 `update_bug(verified=True, ...)`
   - 用户说还有问题 → 继续排查修复

#### 分支 3：用户说“未修复，暂时不解决”

```
好的，我会先记录这个问题，标记为未验证。
等后续修复时再更新根因和解决方案。

当前记录信息：
- 标题：[生成的标题]
- 现象：[用户提供的现象]
- 根因：待补充
- 解决方案：待补充

是否继续记录？
```

## 修复审核（核心流程）

### 情况 A：用户明确说“修好了”

1. 调用 `get_bug_detail(bug_id)` 获取记录
2. **执行「验证流程」**（见下方独立章节）
3. **同步记录信息给用户**：

```
✅ Bug #N 已标记为已验证，记录已更新

**标题**：[标题]
**现象**：[现象]
**根因**：[更新后的准确根因]
**解决方案**：[更新后的最终方案]
**测试用例**：[更新后的测试用例]
**评分**：[各维度分值，如有调整会标注]
**相关路径**：[文件路径]

以上记录已归档到错题集，后续修改相关文件时会自动召回提醒。
```

4. **如果用户提出异议**（如“根因不对”、“方案不准确”等）：
   - 与用户讨论清楚正确的内容
   - 调用 `update_bug()` 更新相应字段（root_cause/solution/test_case 等）
   - 重新同步更新后的记录信息

5. **重要原则**：
   - ✅ **验证通过时必须更新**：根因、解决方案、测试用例必须是最终准确的版本
   - ✅ **分数动态调整**：根据实际修复过程调整 difficulty、emotion 等维度
   - ✅ **避免预估值残留**：初始记录的预估根因/方案必须在验证时替换为准确值

### 情况 B：用户未明确说"修好了"，但切换到新任务

当检测到以下情况时，视为 Bug 已修复：
- ✅ 用户开始讨论完全不同的功能/模块
- ✅ 用户开启了新的开发任务
- ✅ 对话上下文明显脱离了当前 Bug 的排查

**处理流程：**

1. **暂停新任务，先处理遗留 Bug**：
   ```
   在继续新任务前，我先同步一下刚才的 Bug 记录状态。
   
   刚才我们处理的 Bug #N（[标题]），从上下文看似乎已经解决了？
   
   我将根据修复过程更新记录信息并标记为已验证：
   
   **标题**：[标题]
   **现象**：[现象]
   **根因**：[基于最终修复总结的准确根因]
   **解决方案**：[实际采用的有效方案]
   **测试用例**：[验证通过的测试方法]
   
   是否有误？如有问题请告诉我，否则我将继续执行你的新任务。
   ```

2. **等待用户反馈**：
   - **用户无异议、确认或说“已修复”** → 调用 `update_bug(verified=True, ...)` 仅标记为已验证 → 继续新任务
   - **用户指出错误** → 讨论修正 → 执行「验证流程」→ 继续新任务
   - **用户说“还没修好”** → 询问是否先完成修复再继续新任务
   - **用户直接开始新任务的执行**（如提供代码、要求创建文件等）→ 视为默认同意 → 后台静默调用 `update_bug(verified=True, ...)` → 继续执行新任务
   - **超过 3 分钟无响应** → 视为默认同意 → 后台静默调用 `update_bug(verified=True, ...)` → 继续执行新任务

## Bug 复发处理（重要）

**核心原则**：无论初始记录是什么状态（verified=True/False、预估分高低），只要用户反馈“还有问题”，最终验证通过时必须以**实际修复情况**为准更新所有信息。

**处理流程**：

1. **立即打回 `verified=False`**
2. **继续排查修复**，直到用户确认问题解决
3. **执行「验证流程」**（见下方独立章节）

## ✅ 验证通过处理原则

**适用场景**：任何 bug 修复完成后，都需要执行此流程。

**核心原则**：以**实际修复情况**为准更新所有信息，绝不保留预估值。

**处理步骤**：

1. **回顾整个交互过程，评估是否需要调整分数**：
   - **occurrences（出现次数）**：用户是否多次打断、纠正、反馈同样的问题？
     - 如果用户反复说“不对”、“错了”、“还是有问题” → 每次 +1
   - **emotion（用户情绪）**：用户是否有不满、抱怨、急躁的表现？
     - 轻微不满（“怎么又错了”）→ +2
     - 明显不满（“你搞什么”、“太慢了”）→ +3~5
     - 严重不满（“你到底会不会”、“浪费时间”）→ +5~8
   - **difficulty（修复难度）**：实际修复是否比预期复杂？
     - 如果尝试了多次才成功 → +3~5

2. **AI 自动总结最终修复信息**：
   - 回顾整个修复过程，总结**准确的根因**（而非初始预估）
   - 总结**最终有效的解决方案**（包括多次尝试后的正确方案）
   - 评估**实际修复难度**（如果反复修改多次，difficulty 应更高）
   - 确认**测试用例**是否充分

3. **构建完整的 test_case**（如有影响关系需包含被影响功能的验证）：
   - 从上下文获取影响关系（之前调用 `add_impact()` 时已记录）
   - 格式：`[主功能验证]；验证 [被影响路径1] 功能正常；验证 [被影响路径2] 功能正常`
   - 示例："登录后刷新页面验证 session 保持；验证 src/cart/add_to_cart.ts 功能正常"

4. **更新完整信息**：
   - 参考「API 调用指南」调用 `update_bug()`
   - 必须更新：`scores`, `root_cause`, `solution`, `test_case`, `verified=True`

5. **检查分数是否超过阈值**：
   - 如果总分从 <30 变成 >=30，提醒用户此 bug 已升级为高优先级
   - 如果总分 > 50，发出严重警告

**关键要点**：
- ✅ **绝不保留预估值**：初始记录的 root_cause/solution/test_case 只是临时值，必须在最终验证时替换为准确值
- ✅ **反映真实复杂度**：通过 difficulty 累加，让分数反映实际修复难度
- ✅ **重视复发过程**：occurrences 累加反映问题复发的频率
- ✅ **捕捉用户情绪**：emotion 累加反映用户对 AI 表现的满意度

## 🚨 影响关系管理（回归风险）

**核心原则**：影响关系是**客观事实**，一旦用户指出“修改 A 后 B 出问题了”，必须**立即记录**。

### 触发条件

- 用户在修复过程中说：“你改了 XX 后，YY 出问题了”
- 用户明确指出模块间的依赖或副作用

### 处理流程

**发现影响时立即执行**：

参考「API 调用指南」中的「记录影响关系」章节，调用 `add_impact()` 和 `increment_score()`。

**关键步骤**：
1. 调用 `add_impact()` 记录影响关系到独立表
2. 调用 `increment_score(bug_id, "prevention", 3.0~5.0)` 更新主 bug 的 prevention 分数

**影响类型选择：**
- `regression`（回归）：修复一个 bug 导致另一个功能完全失效（最严重）
- `side_effect`（副作用）：修改产生了意外的连带影响
- `dependency`（依赖）：模块间存在强依赖关系，修改需谨慎

**严重程度评估：**（记录在 `bug_impacts` 表的 `severity` 字段）
- 9-10：核心功能完全失效（支付失败、数据丢失）
- 7-8：重要功能受影响（购物车无法添加、登录状态异常）
- 5-6：辅助功能受影响（样式错位、非关键提示缺失）
- 3-4：轻微影响（边缘场景报错、性能轻微下降）
- 0-2：几乎无影响（日志格式变化、注释错误）

**为什么立即更新 prevention？**
- prevention 反映预防价值，有回归风险的 bug 价值更高
- 立即更新确保召回时分数准确，能优先提醒其他 AI

### 例外情况

- 如果用户说"搞错了"，调用 `delete_impact(impact_id)` 并调整 prevention 分数：`increment_score(bug_id, "prevention", -3.0~-5.0)`

## API 调用指南

### 新增 Bug 记录

使用 `scripts/bug_ops.py` 的 `add_bug()` 函数：

```python
from scripts.bug_ops import add_bug

bug_id, score = add_bug(
    title="session 存储未设置持久化",
    phenomenon="登录后刷新页面 session 丢失",
    root_cause=None,
    solution=None,
    test_case="登录后刷新页面验证",
    verified=False,   # 静默记录=True，审核记录=False
    scores={
        "importance": 7,
        "complexity": 5,
        "scope": 4,
        "difficulty": 3,
        "occurrences": 0,
        "emotion": 2,
        "prevention": 6,
    },
    paths=["src/auth/session.ts"],
    tags=["auth", "session"],
    keywords=["session", "cookie"],
    recalls=["auth/*"],
)
```

### 更新 Bug 记录

```python
from scripts.bug_ops import update_bug

update_bug(
    bug_id=42,
    root_cause="准确的根因",
    solution="最终的解决方案",
    test_case="完整的测试用例",
    status="resolved",
    verified=True,
    verified_at="CURRENT_TIMESTAMP",
    verified_by="User"
)
```

### 累加分数

```python
from scripts.bug_ops import increment_score

# 累加出现次数
increment_score(bug_id, "occurrences", 1.0)

# 累加修复难度
increment_score(bug_id, "difficulty", 3.0)

# 累加情绪分
increment_score(bug_id, "emotion", 2.0)
```

### 记录影响关系

```python
from scripts.bug_ops import add_impact

add_impact(
    source_bug_id=42,
    impacted_path="src/cart/add_to_cart.ts",
    impact_type="regression",  # 或 side_effect/dependency
    description="修改 session 持久化导致购物车的用户状态判断失效",
    severity=8,  # 严重程度 0-10
)
```

## 注意事项

- 标题控制在 20 字以内
- 现象和根因要具体，避免模糊描述
- 相关路径用相对于项目根的路径
- autoRecall 优先使用文件路径前缀（如 `auth/`），避免使用太泛的关键词
- 复杂问题不要急着记录，先排查清楚再记
- 根因和解决方案在记录时可以填推断值，后续修复时再更新为准确值
- 小问题自动验证，复杂问题等修复时统一审核
