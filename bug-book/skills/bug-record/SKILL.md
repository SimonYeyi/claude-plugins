# Bug Record Skill

当发现或被指出问题时，将问题的完整信息记录到 bug-book 数据库中。

## Python 脚本执行规则

**【强制要求】禁止 cd 到插件目录！必须在当前项目目录执行 Python 代码。**

- ✅ 正确：保持在项目目录（如 `/home/user/my-project`）执行
- ❌ 错误：cd 到插件目录（如 `~/.claude/plugins/bug-book/scripts`）

如果需要使用插件脚本，使用绝对路径导入：
```python
import sys
sys.path.insert(0, "{CLAUDE_PLUGIN_DIR}/bug-book")
from scripts.bug_ops import add_bug
```

## 判断是否需要记录

### 自动触发的强制信号（无需用户提醒）

当检测到以下任一情况时，**必须立即自动调用本技能**：
- ✅ 用户说"不对"、"错了"、"不是这样"、"回退"等否定词 + 指出具体问题
- ✅ 用户抱怨"之前不是这样的"、"又出现同样的问题"
- ✅ 用户的代码被手动修改或覆盖
- ✅ AI 发现自己引入了回归错误（新功能破坏了旧功能）
- ✅ 排查过程中发现了根因但未提前记录

### 预估分决策逻辑

预估分 = Σ(各维度分值 × 权重)，权重：importance×2.0, complexity×1.5, scope×1.0, difficulty×1.0, occurrences×1.0, emotion×1.5, prevention×2.0

**自动记录**（预估分 < 30）：用户指出明显错误，影响范围小，修复简单。写入时 `verified=True`。

**直接后台记录**（预估分 >= 30）：
- 涉及多个文件或模块
- 根因不明确，需要多轮排查
- 之前已经出现过类似问题
- 用户情绪激动（发火/多次纠正）
- 修复方案复杂
写入时 `verified=False`，修复时再审核。

### 预估分打分示例

| 场景 | importance | complexity | scope | difficulty | emotion | prevention | 总分 | 处理 |
|------|-----------|------------|-------|------------|---------|------------|------|------|
| 笔误：变量名拼错 | 2 | 1 | 1 | 1 | 0 | 1 | **9.5** | 自动记录 |
| 单文件单处bug：按钮样式错位 | 3 | 2 | 2 | 2 | 1 | 3 | **20.0** | 自动记录 |
| 单文件多处bug：同一函数逻辑错误，调用链复杂，需改5处 | 6 | 5 | 3 | 5 | 2 | 5 | **41.0** | 后台记录 |
| 多文件bug：session存储未持久化 | 7 | 5 | 4 | 3 | 2 | 6 | **42.5** | 后台记录 |
| 回归bug：用户第三次抱怨同样问题 | 8 | 6 | 6 | 4 | 3 | 8 | **63.5** | 后台记录 |
| 核心模块内存泄漏 | 9 | 8 | 8 | 7 | 0 | 7 | **66.5** | 后台记录 |

### 决策流程

```
开始评估问题
    │
    ▼
打预估分（参考上表各维度）
    │
    ├── 预估分 < 30 ──→ 自动记录（verified=True）──→ 结束
    │
    └── 预估分 >= 30 ──→ 后台记录（verified=False）──→ 结束
```

## 主动记录模式（用户不在对话上下文中）

当用户主动说“帮我记录一个问题”（这个问题 AI 没有上下文），AI 应该先询问现象，然后立即生成标题，再根据情况引导：

### 第一步：询问现象

```
请描述一下问题的具体表现（现象）是什么？
```

### 第二步：生成标题

用户回答后，**AI 立即根据现象生成标题**（控制在 20 字以内）。

**标题原则：**
- 只从现象中概括，不涉及根因
- 简洁明了，便于快速浏览
- 例如：
  - 现象“刷新页面后需要重新登录” → 标题“刷新页面 session 丢失”
  - 现象“按钮点击没有反应” → 标题“按钮点击无响应”
  - 现象“数据库连接池在高并发时耗尽” → 标题“高并发时连接池耗尽”

**注意：** 标题不影响召回，因为召回会搜索现象、根因、关键词等多个字段。

### 第三步：根据用户回答判断

#### 情况 A：用户只提供了现象

AI 继续询问（此时已有标题）：

```
我理解的问题是：**[生成的标题]**

这个问题是否已经修复？

- 如果已修复：请提供根因和解决方案
- 如果未修复：是否需要我帮你解决这个问题？
```

**用户回复“已修复”：**
```
好的，请提供：
1. **根因**：为什么会这样？
2. **解决方案**：怎么修的？
```

**用户回复“未修复，需要你解决”：**
- AI 开始排查问题
- 找到根因和解决方案后，自动记录（转为自动记录模式）

**用户回复“未修复，暂时不解决”：**
```
好的，我会先记录这个问题，标记为未验证。
等后续修复时再更新根因和解决方案。

当前记录信息：
- 标题：[生成的标题]
- 现象：[用户提供的现象]
- 根因：待补充
- 解决方案：待补充

是否继续记录？
```

#### 情况 B：用户一次性提供了现象、根因、解决方案

- **标题由 AI 根据现象生成**（控制在 20 字以内，不涉及根因）
- 直接调用 `add_bug()` 记录

---

### 第四步：AI 自动处理

- 推断相关路径（`paths`）和 `autoRecall` 模式
- 标签、评分等 AI 自动处理
- 收集完成后直接写入，`verified=False`

### 1. 收集信息

向用户确认以下信息（复杂问题可分步确认）：

```
标题：简述问题（如"用户登录后 session 丢失"）
现象：具体表现（如"刷新页面后需要重新登录"）
根因：为什么会这样（如"session 存储时未设置持久化"）
解决方案：怎么修复的（如"添加 cookie 的 maxAge 配置"）
测试用例：如何验证修复（如"登录后刷新页面，验证 session 保持"）
相关文件/路径：哪些文件相关（如["src/auth/session.ts", "src/middleware/auth.ts"]）
标签：分类标签（如["auth", "session", "cookie"]）
关键词：便于搜索（如["session", "cookie", "maxAge"]）
autoRecall：触发召回的模式（如["auth/*", "session.ts"]）
```

### 2. 评分

根据问题特征给出评分：

| 维度 | 评分依据 |
|------|----------|
| 功能重要性 | 核心功能 8-10，辅助功能 3-7，周边功能 0-2 |
| 逻辑复杂度 | 涉及多模块/异步/并发 7-10，单点逻辑 3-6，简单逻辑 0-2 |
| 影响范围 | 影响整个系统 8-10，影响模块 4-7，影响单个文件 0-3 |
| 修复难度 | 需要大重构 8-10，需要改多个文件 5-7，单文件修复 0-4 |
| 出现次数 | 首次出现 +0，重复出现 +1（可累加） |
| 用户情绪 | 被纠正 +2，发火 +5 |
| 预防价值 | 高频场景 +8-10，偶发场景 +3-7，低频场景 +0-2 |

### 3. 写入数据库

使用 `scripts/bug_ops.py` 的 `add_bug()` 函数：

```python
from scripts.bug_ops import add_bug

bug_id, score = add_bug(
    title="session 存储未设置持久化",
    phenomenon="登录后刷新页面 session 丢失",
    root_cause=None,
    solution=None,
    test_case="登录后刷新页面验证",
    verified=False,  # 小问题为 True，复杂问题为 False
    scores={
        "importance": 7,
        "complexity": 5,
        "scope": 4,
        "difficulty": 3,
        "occurrences": 0,
        "emotion": 2,
        "prevention": 6,
    },
    paths=["src/auth/session.ts"],
    tags=["auth", "session"],
    keywords=["session", "cookie"],
    recalls=["auth/*"],
)
```

### 4. 通知用户（复杂问题）

对于需要审核的问题，直接后台记录后告知用户：

```
已后台记录为 Bug #N，当前未验证。
当功能修复后请告知我，我来验证并更新记录。
```

### 5. 修复审核（核心流程）

#### 情况 A：用户明确说"修好了"

1. 调用 `get_bug_detail(bug_id)` 获取记录
2. **直接更新为已验证**：调用 `update_bug(verified=True, verified_at="CURRENT_TIMESTAMP", verified_by="User")`
3. **同步记录信息给用户**：

```
✅ Bug #N 已标记为已验证

**标题**：[标题]
**现象**：[现象]
**根因**：[根因]
**解决方案**：[方案]
**测试用例**：[测试用例]
**评分**：[各维度分值]
**相关路径**：[文件路径]

以上记录已归档到错题集，后续修改相关文件时会自动召回提醒。
```

4. **如果用户提出异议**（如"根因不对"、"方案不准确"等）：
   - 与用户讨论清楚正确的内容
   - 调用 `update_bug()` 更新相应字段（root_cause/solution/test_case 等）
   - 重新同步更新后的记录信息

#### 情况 B：用户未明确说"修好了"，但切换到新任务

当检测到以下情况时，视为 Bug 已修复：
- ✅ 用户开始讨论完全不同的功能/模块
- ✅ 用户开启了新的开发任务
- ✅ 对话上下文明显脱离了当前 Bug 的排查

**处理流程：**

1. **暂停新任务，先处理遗留 Bug**：
   ```
   在继续新任务前，我先同步一下刚才的 Bug 记录状态。
   
   刚才我们处理的 Bug #N（[标题]），从上下文看似乎已经解决了？
   
   我将把它标记为已验证并归档，以下是记录信息：
   
   **标题**：[标题]
   **现象**：[现象]
   **根因**：[推断的根因]
   **解决方案**：[实际采用的方案]
   
   是否有误？如有问题请告诉我，否则我将继续执行你的新任务。
   ```

2. **等待用户反馈**：
   - **用户无异议或确认** → 调用 `update_bug(verified=True, ...)` → 继续新任务
   - **用户指出错误** → 讨论修正 → 更新记录 → 继续新任务
   - **用户说"还没修好"** → 询问是否先完成修复再继续新任务
   - **用户直接开始新任务的执行**（如提供代码、要求创建文件等）→ 视为默认同意 → 后台静默标记为已验证 → 继续执行新任务
   - **超过 3 分钟无响应** → 视为默认同意 → 后台静默标记为已验证 → 继续执行新任务

## 记录后提醒

记录完成后，提醒用户关键信息：

- 这个问题被记录为 Bug #N，总分 {score}
- autoRecall 匹配模式：`[模式]`
- 整理错题集时会优先考虑这个问题

## 注意事项

- 标题控制在 20 字以内
- 现象和根因要具体，避免模糊描述
- 相关路径用相对于项目根的路径
- autoRecall 优先使用文件路径前缀（如 `auth/`），避免使用太泛的关键词
- 复杂问题不要急着记录，先排查清楚再记
- 根因和解决方案在记录时可以填推断值，后续修复时再更新为准确值
- 小问题自动验证，复杂问题等修复时统一审核
