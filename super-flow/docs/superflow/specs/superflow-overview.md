# 超级生产线（SuperFlow）— 设计文档

## 一、插件概述

通过 `/superflow` 斜杠命令或任何功能性开发请求触发，拦截所有"构建/开发/实现"类需求。

**两种模式**：
- 创意模式：创意由创意Agent提出，brainstorming后由创意Agent确认SPEC
- 产品模式：创意由用户提出，brainstorming后由用户确认SPEC

**统一流程**：SPEC确认后，后续流程（架构 → 开发 → 测试 → 评审）均为全自动，无需用户介入

## 二、入口分支

| 场景                            | 触发 |
|-------------------------------|------|
| `/superflow`（无任何参数）           | 直接进入**创意模式** |
| `/superflow <有主题但无明确功能细节>`    | 询问选择：创意模式 / 产品模式 |
| `/superflow <有主题有细节> 或 <模糊主题>` | 直接进入**产品模式**（brainstorm 流程，探讨需求） |

### 各阶段Agent与内嵌审查Agent

| 阶段 | 主干Agent | 内嵌审查Agent | 说明 |
|------|--------|-------------|------|
| 创意Agent | 创意评审团 | 创新性+可行性+商业价值，3或5个并行评审，失败内循环修复 |
| 产品Agent | SPEC审查Agent | 验证SPEC执行创意，失败打回产品Agent修复 |
| 架构Agent | 计划评审Agent | 验证计划完整性，失败打回架构Agent修复 |
| 开发Agent | 实现评审团（3实例并行） | 完整性+代码质量+安全，失败打回开发Agent修复 |
| 测试Agent | 测试评审Agent | 验证测试覆盖SPEC，失败打回测试Agent修复 |

**SPEC确认统一规则**：谁提出创意，谁确认SPEC
- 创意模式：创意由创意Agent提出 → 创意Agent确认
- 产品模式：创意由用户提出 → 用户确认

## 四、评审/审查循环规则

**内循环机制**：
- 每个主干Agent完成工作后，启动自己的**内嵌审查Agent**进行审查
- 审查有意见时，由该主干Agent直接处理问题，形成"工作→审查→修复→再审查"闭环
- 审查通过后，由该主干Agent通知主控，主控再启动下一个主干Agent

**主干Agent处理评审意见的职责**：
1. **独立思考**：对每条评审意见进行独立判断，不盲目接受
2. **有问题则修改**：如果认为评审意见确实有理，则按意见修改
3. **坚持己见并反馈**：如果认为评审意见有问题，可以坚持己见，但必须将不修改的具体理由反馈给评审Agent

**评审Agent处理反馈意见的职责**：
1. **重新核实**：对主干Agent已修复的问题重新核实
2. **重新思考**：对主干Agent反馈的不修改意见，重新思考是否同意
   - 如果同意：不追究，记录在案
   - 如果不同意：给出明确的不同意理由，打回主干Agent

**双向沟通循环**：
```
评审Agent给出意见 → 主干Agent独立思考
    │
    ├──→ 认为有理 → 修改 → 反馈给评审Agent重新核实
    │
    └──→ 认为无理 → 坚持己见 + 反馈不修改理由 → 评审Agent重新思考
                                                                │
                                                    ┌───────────┴───────────┐
                                                    │                       │
                                                同意（记录）          不同意（打回）
```

**评审意见不一致时**：
- 主干Agent与内嵌审查Agent之间可以直接通信
- 分歧采用**双向沟通机制**解决，而非简单投票
- 循环直到达成共识，或升级主控裁断
- **评审最大重试次数**：每个内循环最多重试 **5 次**，5 次后仍有分歧则交由主控决断

**主控职责**：
- 按顺序启动各主干Agent
- 监控各阶段内循环直到通过
- 协调阶段之间的交接
- 内循环5次后仍分歧时裁断

**终止条件**：全部问题修复，所有评审/审查 agent 全部通过，且主控确认。

**主控最终确认检查表**：
- [ ] SPEC 文档已确认并保存
- [ ] 实现计划已评审通过
- [ ] 实现评审团全部通过（完整性+代码质量+安全）
- [ ] 测试用例完整覆盖 SPEC 验收标准（覆盖率审查通过）
- [ ] 测试报告已生成并保存
- [ ] 所有产出物已保存至对应目录
- [ ] 流程无遗漏环节

**产出物清单**：
- Creative Brief（仅创意模式）
- SPEC.md
- 实现计划
- 代码实现
- 测试用例文档 + 单元测试代码
- **测试报告**
- 所有产出物已保存至对应目录

**完成定义**：测试评审通过后，主控对照检查表逐项确认所有产出物齐全，正式宣告流程完成。

**产出物流程**：
1. 测试Agent执行测试 → 生成**测试报告**
2. 测试评审Agent审查测试用例覆盖率+质量精度
3. 覆盖率和质量达标 → 主控确认所有产出物 → 流程完成
4. 覆盖率和质量达标但有遗留问题 → 记录在案 → 主控确认 → 流程完成
5. 5次循环后仍有分歧 → 主控决断 → 根据决断结果处理

## 六、Agent功能大纲

### 创意Agent

- **定位**：CEO/高级产品战略官
- **输入**：用户需求/主题（可为空，需要自主创新）
- **输出**：Creative Brief（创意说明书）
- **步骤**：
  1. 分析用户需求和目标（如有）
  2. 研究市场、竞品、用户
  3. 提出创意方向和差异化
  4. 输出Creative Brief草稿
  5. 进入**创意评审团内循环**评审（无需用户确认）
  6. 评审通过后**写入** `docs/superflow/creatives/`
- **要求**：
  - 创意需有创新性、可行性、商业价值
  - 评审通过后才能写入，评审期间不得提前写入
  - brainstorming中如采纳产品agent建议，需更新Creative Brief

### 创意评审团

- **定位**：多视角评审委员会
- **输入**：Creative Brief
- **输出**：评审意见（通过/不通过）
- **实例数**：3 个（子功能创意）或 5 个（项目立项创意）
- **步骤**：
  1. 每个实例从全部视角审查（创新性 + 可行性 + 商业价值）
  2. 3 个或 5 个实例并行评审
  3. 与创意Agent进行双向讨论，创意Agent可申辩或修改
  4. 汇总评审结果，通过双向沟通达成共识
- **要求**：每个实例评估全部视角；评审有分歧时通过双向沟通解决，5次循环后仍有分歧交由主控决断

### 产品Agent

- **定位**：需求转化专家
- **输入**：Creative Brief 或 用户原始需求
- **输出**：SPEC.md
- **步骤**：
  1. brainstorming对话（创意模式：一次全问；产品模式：一次一问、多选题优先、提出方案给建议）
  2. 整合brainstorming所有内容在上下文中输出
  3. **与创意提出者确认SPEC（必须先确认才能写入）**
  4. 确认后写入 `docs/superflow/specs/`
- **要求**：
  - SPEC需非技术人员可读，不含代码细节
  - SPEC必须包含明确的、详细的验收标准，以便测试Agent和测试评审Agent评审

### SPEC审查Agent

- **定位**：规格书审查
- **输入**：SPEC.md + Creative Brief 或 brainstorming对话记录
- **输出**：SPEC审查报告
- **步骤**：
  1. 对照Creative Brief逐项检查SPEC（创意模式）
  2. 或对照brainstorming对话记录逐项检查SPEC（产品模式）
  3. 验证每条需求是否有对应SPEC实现
  4. 输出审查结论
- **要求**：确保SPEC完全执行brainstorming结果，无遗漏无偏差

### 架构Agent

- **定位**：架构设计专家
- **输入**：SPEC.md
- **输出**：实现计划（含架构设计）
- **步骤**：
  1. 分析SPEC需求
  2. 设计模块划分、接口、数据流
  3. 确定设计模式
  4. 分解Task（按实现顺序）
  5. 编写Spec覆盖检查表
- **要求**：
  - OOP语言须遵循SOLID等设计规范
  - 遇到可复用组件时先思考能不能复用，能则直接复用，有差异时适配后再复用
  - 架构设计的意图和在特殊情况下的权衡取舍需在计划书中说明，以便开发者理解和实现评审团评审

### 实现计划评审Agent

- **定位**：实现计划审查
- **输入**：实现计划文档
- **输出**：评审报告
- **步骤**：
  1. 验证计划完整性
  2. 验证架构合理性
  3. 检查Task分解是否合理
- **要求**：
  - 发现问题时打回架构Agent重做，直到通过

### 开发Agent

- **定位**：开发执行者
- **输入**：实现计划
- **输出**：代码
- **步骤**：
  1. 按计划执行开发
  2. 遇到问题自行修复
  3. 完成开发后交测试Agent验证
- **要求**：
  - 严格按照计划执行
  - 代码需符合SOLID原则、代码规范，以便通过实现评审团评审

### 实现评审团

- **定位**：终审视角委员会
- **输入**：代码 + SPEC.md
- **输出**：审查报告（完整性+代码质量+安全）
- **实例数**：3 个
- **步骤**：
  1. 每个实例从全部视角审查（完整性 + 代码质量 + 安全）
  2. 3 个实例并行评审
  3. 与开发Agent进行双向讨论，开发Agent可申辩或修改
  4. 汇总评审结果，通过双向沟通达成共识
- **要求**：在测试评审之前进行；通过双向沟通解决分歧，5次循环后仍有分歧交由主控决断

### 测试Agent

- **定位**：测试专家
- **输入**：SPEC.md + 代码
- **输出**：测试用例文档 + 单元测试代码
- **步骤**：
  1. 产出逻辑测试用例文档
  2. 产出非逻辑测试用例文档
  3. 编写单元测试
  4. 执行测试
- **要求**：
  - 逻辑测试+非逻辑测试需100%覆盖SPEC的验收标准
  - **逻辑测试用例**：可通过自动化脚本/工具完成验证的部分（如 API 响应、数据处理、业务逻辑计算等）
  - **非逻辑测试用例**：需人工介入验证的部分（如 UI 交互、系统硬件、Agent Skills 行为、跨系统集成等）
  - 逻辑测试可自动化执行，非逻辑测试需人工验证
  - **单元测试执行结果处理**：
    - 测试不通过时，若原因是**测试代码错误** → 打回测试Agent修改测试代码
    - 测试不通过时，若原因是**功能验证不通过** → 打回开发Agent修复功能，修复后重新进入测试Agent

### 测试评审Agent

- **定位**：测试覆盖率 + 质量精度审查
- **输入**：SPEC.md + 测试用例文档 + 测试报告
- **输出**：测试评审报告
- **步骤**：
  1. 对照SPEC逐项检查测试用例覆盖
  2. 验证逻辑测试和非逻辑测试区分清晰
  3. 检查每条验收标准是否有对应测试用例
  4. 评估测试用例质量精度（Precision + Detail + Completeness + Clarity + Measurability）
  5. 输出测试评审结论
- **要求**：确保测试用例完整覆盖SPEC验收标准且质量达标；发现问题时打回测试Agent修改，直到通过

## 七、创意 Agent 与 产品 Agent 交接与流程规则

**brainstorming 通信机制：**
- 产品Agent提出问题 → 主控转发 → 创意Agent/用户
- 创意Agent/用户回答 → 主控转发 → 产品Agent
- 循环直到brainstorming完成 → 整合所有内容 → **必须先给创意提出者确认 → 确认后才能写入**

**关键原则：**
- 创意 Agent 决定"做什么、为什么做" — 产品 Agent 决定"怎么做、功能细节"
- 如果 SPEC 草稿与创意方向有偏差，创意 Agent 有权要求修正
- **先确认再写入，不得先写入再确认**

**SPEC 文档要求**：
- SPEC 是通用文档，非技术人员也能读懂
- 不得包含代码片段、API 实现细节等技术细节
- 只描述功能需求、用户交互、验收标准

## 八、文件路径

按功能类型分目录，命名格式：`YYYY-MM-DD-feature-name-xxx.md`，整个流程共用同一个 feature-name。

**所有文档写入文件时，必须提示输出的文件路径**

```
docs/superflow/
├── specs/          # SPEC 文档
├── plans/          # 实现计划
├── creatives/      # 创意文档
└── tests/          # 测试用例（logic-tests.md, manual-tests.md, test-report.md）
```

## 十、审查顺序与视角

**审查顺序**：
1. **SPEC 审查**：验证 SPEC 是否完全执行 Creative Brief 的创意（SPEC 确认后，开发前）
2. **计划评审**：架构 Agent 生成计划后，计划评审 Agent 验证计划完整性及架构合理性，**评估SPEC可实现性**（计划完成后，开发前）
3. **实现评审团**：验证实现完整性 + 代码质量 + 安全（3个并行实例，开发完成后）
4. **测试评审**：验证测试用例覆盖率 + 质量精度（Precision + Detail + Completeness）
5. **主控确认**：所有审查通过后，主控最终确认

| 视角 | 检查内容 |
|------|---------|
| **SPEC** | SPEC 是否完全执行 Creative Brief 的创意 |
| **架构** | 设计模式、模块边界、性能考虑、可扩展性、**SPEC可实现性** |
