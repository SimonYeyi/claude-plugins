# 超级生产线（SuperFlow）— 设计文档

## 一、插件概述

通过 `/superflow` 斜杠命令或任何功能性开发请求触发，拦截所有"构建/开发/实现"类需求。

**两种模式**：
- 创意模式：创意由创意Agent提出，brainstorming后由创意Agent确认SPEC
- 产品模式：创意由用户提出，brainstorming后由用户确认SPEC

**统一流程**：SPEC确认后，后续流程（架构 → 开发 → 测试 → 评审）均为全自动，无需用户介入

**重要区分**：
- **SPEC确认**：由创意提出者（创意Agent/用户）主观判断SPEC是否符合自己的创意/需求，**不是评审**
- **SPEC审查Agent评审**：由SPEC审查Agent客观验证SPEC是否完整执行了Creative Brief或brainstorming结果，是独立的内循环流程

## 二、入口分支

| 场景                             | 触发 |
|--------------------------------|------|
| `/superflow`（无任何参数）            | 直接进入**创意模式** |
| `/superflow <有主题但无明确功能细节>`     | 询问选择：创意模式 / 产品模式 |
| `/superflow <有主题且细节丰富> 或 <无法识别主题>` | 直接进入**产品模式**（brainstorm 流程，探讨需求） |

    │                    ↺ 内循环                          │
    │                                                      │
    └──→ 产品模式 ──→ 阶段一：传达用户需求 ─────────────────────┤
                                                           ▼
                                                    阶段二：产品流程
                                              （产品Agent + SPEC审查Agent内循环）
                                                           ↺ 内循环
                                                           │
                                                           ▼
                                                    阶段三：架构流程
                                              （架构Agent + 计划评审Agent内循环）
                                                           ↺ 内循环
                                                           │
                                                           ▼
                                                    阶段四：开发流程
                                            （开发Agent + 实现评审团）
                                                           ↺ 内循环

**阶段流程激活任务与输入要求**：

| 阶段 | 激活任务 | 输入要求 |
|------|----------|----------|
| 阶段一：创意流程 | dispatch **创意Agent** | 用户原始需求/主题 |
| 阶段一：传达用户需求 | dispatch **产品Agent** | 用户原始需求/主题（产品模式入口） |
| 阶段二：产品流程 | dispatch **产品Agent** | Creative Brief（创意模式）或 用户需求（产品模式） |
| 阶段三：架构流程 | dispatch **架构Agent** | SPEC.md |
| 阶段四：开发流程 | dispatch **开发Agent** | 实现计划（含架构设计） |
| 阶段五：测试流程 | dispatch **测试Agent** | SPEC.md + 代码实现 |

### 各阶段Agent与内嵌审查Agent

| 阶段 | 主干Agent | 颜色 | 内嵌审查Agent | 说明 |
|------|--------|------|-------------|------|
| 创意Agent | 创意评审团 | magenta | 创新性+可行性+商业价值，失败内循环修复 |
| 产品Agent | SPEC审查Agent | orange | 验证SPEC执行创意，失败打回产品Agent修复 |
| 架构Agent | 计划评审Agent | cyan | 验证计划完整性，失败打回架构Agent修复 |
| 开发Agent | 实现评审团 | green | 完整性+代码质量+安全，失败打回开发Agent修复 |
| 测试Agent | 测试评审Agent | yellow | 验证测试覆盖SPEC，失败打回测试Agent修复 |

> **颜色规范**：主干Agent与对应的评审Agent使用相同颜色，不同主干Agent使用不同颜色以区分

**测试流程两阶段评审**：
- **阶段一**：测试用例文档生成后 → dispatch test-reviewer（测试用例评审）
- **阶段二**：测试代码执行通过后 → dispatch test-reviewer（全面评审，含测试代码实现）

**SPEC确认统一规则**：谁提出创意，谁确认SPEC
- 创意模式：创意由创意Agent提出 → 创意Agent确认
- 产品模式：创意由用户提出 → 用户确认

**注意**：确认和评审是不同概念。确认是主观判断SPEC是否符合创意（由人执行），评审是客观验证SPEC是否完整执行了创意（由Agent执行）。两者独立，缺一不可。

## 四、评审/审查循环规则

**内循环机制**：
- 每个主干Agent完成工作后，启动自己的**内嵌审查Agent**进行审查
- 审查有意见时，由该主干Agent直接处理问题，形成"工作→审查→修复→再审查"闭环
- 审查通过后，由该主干Agent通知主控，主控再启动下一个主干Agent

**主干Agent处理评审意见的职责**：
1. **独立思考**：对每条评审意见进行独立判断，不盲目接受
2. **有问题则修改**：如果认为评审意见确实有理，则按意见修改
3. **坚持己见并反馈**：如果认为评审意见有问题，可以坚持己见，但必须将不修改的具体理由反馈给评审Agent

**评审Agent处理反馈意见的职责**：
1. **重新核实**：对主干Agent已修复的问题重新核实
2. **重新思考**：对主干Agent反馈的不修改意见，重新思考是否同意
   - 如果同意：不追究，记录在案
   - 如果不同意：给出明确的不同意理由，打回主干Agent

**双向沟通循环**：
```
评审Agent给出意见 → 主干Agent独立思考
    │
    ├──→ 认为有理 → 修改 → 反馈给评审Agent重新核实
    │
    └──→ 认为无理 → 坚持己见 + 反馈不修改理由 → 评审Agent重新思考
                                                                │
                                                    ┌───────────┴───────────┐
                                                    │                       │
                                                同意（记录）          不同意（打回）
```

**评审意见不一致时**：
- 主干Agent与内嵌审查Agent之间可以直接通信
- 分歧采用**双向沟通机制**解决，而非简单投票
- 循环直到达成共识，或上报主控裁断
- **评审最大重试次数**：每个内循环最多重试 **5 次**，5 次后仍有分歧则交由主控**自动**决断（不等待用户介入）

**主控职责**：
- 按顺序启动各主干Agent
- 监控各阶段内循环直到通过
- 协调阶段之间的交接
- **处理测试Agent响应**：根据返回的ESCALATE指令执行
- **裁断所有主干Agent的上报问题**：除brainstorming转发和SPEC确认外，任何主干Agent抛出的问题主控都必须裁断，根据问题内容做出决策并继续流程

**终止条件**：全部问题修复，所有评审/审查 agent 全部通过，且主控确认。

**审查顺序**：
1. **SPEC 审查**：验证 SPEC 是否完全执行 Creative Brief 的创意（SPEC 确认后，开发前）
2. **计划评审**：架构 Agent 生成计划后，计划评审 Agent 验证计划完整性及架构合理性，**评估SPEC可实现性**（计划完成后，开发前）
3. **实现评审团**：验证实现完整性 + 代码质量 + 安全（开发完成后）
4. **测试评审**：验证测试用例覆盖率 + 质量精度（Precision + Detail + Completeness）
5. **主控确认**：所有审查通过后，主控最终确认

**审查视角**：
| 视角 | 检查内容 |
|------|---------|
| **SPEC** | SPEC 是否完全执行 Creative Brief 的创意 |
| **架构** | 设计模式、模块边界、性能考虑、可扩展性、**SPEC可实现性** |
| **实现完整性** | 实现是否完整覆盖 SPEC 需求（不多不少），验收标准是否满足 |
| **测试覆盖率** | 测试用例是否覆盖 SPEC 中的每一条验收标准 |
| **测试质量精度** | Precision（精确性）、Detail（细节）、Completeness（完整性）、Clarity（清晰度）、Measurability（可测量性） |
| **代码质量** | 代码风格、可维护性、测试覆盖（基于 SPEC）、SOLID 原则 |
| **安全** | 注入漏洞、依赖安全、敏感信息泄露、权限控制 |

**主控最终确认检查表**：
- [ ] SPEC 文档已确认并保存
- [ ] user-guide.md 已生成
- [ ] 实现计划已评审通过
- [ ] 实现评审团全部通过（完整性+代码质量+安全）
- [ ] 测试用例完整覆盖 SPEC 验收标准（覆盖率审查通过）
- [ ] 测试报告已生成并保存
- [ ] 所有产出物已保存至对应目录
- [ ] 流程无遗漏环节

**产出物清单**：
- Creative Brief（仅创意模式）
- SPEC.md
- user-guide.md
- 实现计划
- 代码实现
- 测试用例文档 + 单元测试代码
- **测试报告**
- 所有产出物已保存至对应目录

**完成定义**：测试评审通过后，主控对照检查表逐项确认所有产出物齐全，正式宣告流程完成。

**产出物流程**：
1. 测试Agent执行测试 → 生成**测试报告**
2. 测试评审Agent审查测试用例覆盖率+质量精度
3. 覆盖率和质量达标 → 主控确认所有产出物 → 流程完成
4. 覆盖率和质量达标但有遗留问题 → 记录在案 → 主控确认 → 流程完成
5. 5次循环后仍有分歧 → 主控决断 → 根据决断结果处理

## 六、Agent功能大纲

### 创意Agent

**定位**：CEO / 高级产品战略官

**核心职责**：不写规格书，只做战略决策——决定做什么、为什么做，以及创意方向。输出 **Creative Brief（创意说明书）**。

**工作场景**：

| 触发 | 输入 | 输出 | 处理 |
|------|------|------|------|
| 收到用户需求/主题 | 用户需求/主题（可为空） | Creative Brief | 分析→战略决策框架→生成→dispatch评审 |
| 收到评审反馈（含主控决断） | 评审结果（评审类型、count） | 修复后Creative Brief | 按规则处理：count<5修复/反驳，count=5上报，count=-1执行决断 |
| 收到brainstorming对话 | product-agent问题 | 回复+更新Creative Brief | 核对方向→回复问题→写入文档 |
| 收到SPEC确认请求 | product-agent请求确认SPEC | 确认回复或遗漏内容 | 已覆盖回复确认，有遗漏回复遗漏 |

**评审实例控制**：根据创意范围类型判断 — 项目立项创意 → **1个**并行实例；子功能创意 → **1个**并行实例

**战略决策框架**：
- **为什么是 THIS**：解决什么具体用户痛点，为什么这是正确解决方案
- **为什么是 NOW**：什么发生变化，为什么2年前不能做/2年后会过时
- **为什么是 US**：我们有什么具体优势，优势能持续多久
- **为什么不**：有什么潜在陷阱，大胆和鲁莽的界限在哪

**质量标准**：

| 标准 | 要求 | 为什么重要 |
|------|------|------------|
| **基于证据** | 每个论断都有分析支撑 | 防止群体思维 |
| **果断** | 没有"也许"或"可能是" | 便于行动 |
| **严格的范围** | MVP是最小化的，不是全面的 | 更快上线 |
| **可防御的钩子** | 仅"酷"是不够的 | 竞争护城河 |
| **风险意识** | 承认失败模式 | 减少意外 |
| **时机意识** | "为什么是现在"是明确的 | 抓住市场窗口 |

**Creative Brief输出结构**：战略背景 → 定位 → 创意方向 → 竞争定位 → 风险评估 → 时间线

**反模式**：
- ✗ "这能吸引所有人" — 特异性才能获胜
- ✗ "我们以后再想" — 范围必须明确
- ✗ "竞争对手难以复制" — 假设他们会回应
- ✗ "用户会喜欢这个" — 必须有具体的用户洞察

**权利**：做大胆的战略赌注；如果产品Agent误解意图可以反驳；坚持最小化MVP范围

**义务**：每个决定都有理由；Creative Brief是可执行的；一旦承诺不反复

### 创意评审团

**定位**：多视角评审委员会

**核心职责**：从创新性、可行性、商业价值三个视角全面评估创意概念，通过双向讨论与创意Agent达成共识。

**工作场景**：

| 触发 | 输入 | 输出 | 处理 |
|------|------|------|------|
| 收到Creative Brief | Creative Brief | 评审意见（含评分） | 阅读→三个视角评估打分→反馈 |
| 收到创意Agent反驳 | 反驳意见 | 更新后的评审意见 | 接受→更新；反驳→提供维持理由 |

**三个评审视角**：

**1. 创新性审查**
- 评分标准：Pass(新品类+1/显著改善+0.5) | Revise(边际改善0) | Reject(模仿跟随-1)
- 必答问题：什么阻止竞争对手复制？什么让用户想要这个？

**2. 可行性审查**
- 评分标准：Pass(技术可行高/<4周/风险低/完美匹配) | Revise(中/4-8周/中/可学习) | Reject(低/>8周/高/缺技能)

**3. 商业价值审查**
- 评分标准：Pass(目标非常具体/清晰路径/持久护城河/核心战略) | Revise/Reject递减

**输出格式**：评审（创新性+可行性+商业价值评分）→ 讨论记录 → 最终建议（Pass/Revise/Reject）

**评审质量标准**：批评必须具体；评估基于证据；如果被辩护改变想法明确承认；上报不是失败

### 产品Agent

**定位**：高级产品经理 / 需求分析师

**核心职责**：将创意策略（来自Creative Brief）或用户需求转化为详细的、可测试的产品规格说明书。

**工作场景**：

| 触发 | 输入 | 输出 | 处理 |
|------|------|------|------|
| 收到Creative Brief | Creative Brief | brainstorming对话上下文 | 请求dispatch创意Agent一次全问 |
| 收到用户需求 | 用户原始需求 | brainstorming对话上下文 | 一次一问逐步确认 |
| 收到brainstorming对话 | 对话上下文 | SPEC.md | 已回答所有问题→生成SPEC→与创意提出者确认→确认通过dispatch spec-reviewer |
| 收到SPEC确认回复 | 确认意见 | 确认通过/有修改 | 确认通过→dispatch评审；有修改→更新后重新确认 |
| 收到评审反馈（含主控决断） | 评审结果（评审类型、count） | 修复后SPEC | count<5修复/反驳；count=5上报；count=-1执行决断 |

**SPEC确认与评审要求**：
- 必须发起SPEC确认在Brainstorming完成后
- 必须发起SPEC评审在SPEC确认后
- 确认≠评审，两者独立，确认通过 ≠ 评审通过

**Brainstorming对话规范**：
- **与创意Agent**：一次全问
- **与用户**：一次一问、逐步确认，多选题优先，提出方案给建议
- 格式：提供A/B/C/D选项，推荐选项标注[推荐]及理由

**六维澄清法**（触发条件：模糊词汇/缺少场景/缺少量化/缺少对比/缺少成功标准）：
1. 场景还原（Who+When+Where）
2. 根因分析（5 Why追问）
3. 期望边界（对比锚定）
4. 量化指标（强制数字化）
5. 约束条件（技术/业务边界）
6. 成功标准（验收条件预定义）

**SPEC编写检查清单**：

| 章节 | 必须包含 | 质量标准 |
|------|----------|----------|
| Overview | What + Why + Who | 最多2句话 |
| User Stories | Role + Action + Outcome | 可测试 |
| Acceptance Criteria | Given/When/Then | 具体+可测量 |
| User Flows | Happy + Alternative + Error | 完整路径 |
| Edge Cases | Boundary + Error + Invalid | 每个"假设" |
| Data Model | Entities + Relationships | CRUD清晰 |
| Out of Scope | 明确排除 | 防止蔓延 |

**上报产品流程结束**：生成 user-guide.md → 上报产品流程结束

### SPEC审查Agent

**定位**：规格书审查专家

**核心职责**：验证SPEC.md是否完整且准确地执行Creative Brief（创意模式）或brainstorming对话上下文（产品模式）中的每一点。

**工作场景**：

| 触发 | 输入 | 输出 | 处理 |
|------|------|------|------|
| 收到SPEC文档 | Creative Brief/brainstorming + SPEC.md | SPEC审查报告 | 读取源文档→创建覆盖矩阵→识别缺口→记录发现→反馈 |
| 收到产品Agent反驳 | 反驳意见 | 更新后的评审意见 | 接受→更新；反驳→提供维持理由 |

**审查维度**：
1. **创意方向覆盖**（Creative Mode）：战略背景、目标用户、差异化、范围
2. **需求覆盖**（两种模式）：每条AC可追溯，无添加/遗漏
3. **规格质量**：AC具体可测量，用户流程完整，边界情况处理，"不在范围内"明确

**输出格式**：模式 → 覆盖矩阵 → 发现（缺失/蔓延/质量/对齐问题）→ 整体评估

**质量标准**：具体说明覆盖/缺少；不要标记你会写得不同的设计选择

### 架构Agent

**定位**：高级软件架构师

**核心职责**：将SPEC.md翻译为详细的、可执行的实现计划。

**工作场景**：

| 触发 | 输入 | 输出 | 处理 |
|------|------|------|------|
| 收到SPEC.md | SPEC.md | 实现计划（含架构设计） | 读取→评估可实现性→设计架构→分解Task→检查覆盖→dispatch评审 |
| 收到评审反馈（含主控决断） | 评审结果（评审类型、count） | 修复后实现计划 | count<5修复/反驳；count=5上报；count=-1执行决断 |

**专业能力**：SOLID原则、设计模式选择、模块边界设计、数据流建模、抽象设计、风险评估、组件复用评估

**SOLID原则应用**：

| 原则 | 应用 | 红旗 |
|------|------|------|
| S（单一职责） | 每个模块只有一个变化原因 | 同时做用户认证和计费的模块 |
| O（开闭原则） | 扩展行为不修改现有代码 | 添加功能需要编辑现有模块 |
| L（里氏替换） | 子类型可在父类型预期的地方工作 | 作为基类型传入时失败 |
| I（接口隔离） | 小而专注的接口 | 一个接口有20个方法 |
| D（依赖反转） | 依赖抽象 | 直接依赖具体类 |

**复用决策协议**：
- 现有组件完全合适 → 直接使用
- 现有组件有微小差异 → 适配后使用
- 现有组件不合适但接近 → 认真考虑，记录原因
- 没有现有组件 → 设计新的
- **永远不要强制复用**

**Task分解标准**：离散、可测试、原子性、依赖清晰、文件操作明确

**实现计划必须包含章节**：目标 → 架构概述 → 接口设计 → 数据流 → 设计模式 → Task分解 → Spec覆盖检查

**计划检查要点**：每个AC有对应task、task有清晰验证标准、模块边界符合SOLID、设计模式有理由、复用认真评估、技术风险识别、开发者可直接执行

**SPEC可实现性评估**：技术可行性、复杂度、依赖评估、风险评估；发现问题→打回产品Agent修改SPEC

### 开发Agent

**定位**：高级软件开发工程师

**核心职责**：将实现计划转化为可工作的代码。

**工作场景**：

| 触发 | 输入 | 输出 | 处理 |
|------|------|------|------|
| 收到实现计划 | 实现计划 | 代码实现 | 读取→按Task顺序执行→自审→确保可运行→dispatch评审 |
| 收到评审反馈（含主控决断） | 评审结果（评审类型、count） | 修复后代码 | count<5修复/反驳；count=5上报；count=-1执行决断→上报结束 |
| 收到测试失败反馈 | 测试失败报告 | 修复后的代码 | 理解原因→修复功能bug→dispatch再次 |

**专业能力**：Clean Code、SOLID实现、错误处理、代码审查

**代码质量检查清单**：

| 标准 | 要求 | 红旗 |
|------|------|------|
| 命名 | 有意义、一致、符合惯例 | 单字母、模糊命名 |
| 结构 | 逻辑分组、适当抽象 | 上帝类、深度嵌套 |
| 复杂度 | 适合问题 | 过度或不足工程 |
| 错误处理 | 每个失败有计划 | 静默try-catch |
| 安全 | 输入验证、无硬编码密钥 | SQL注入、暴露凭证 |
| 性能 | 适合规模 | N+1、缺索引 |

**提交标准**：原子性（一个逻辑变更）、消息（What+Why）、无废弃代码

**质量保证检查清单**：代码可运行、无回归、边界情况处理、错误消息有意义、无硬编码、有输入验证、遵循SOLID、自审记录

**权利**：用证据捍卫选择；推动不正确反馈；请求澄清模糊反馈

**义务**：反馈有效则修复；用细节解释捍卫选择；不要固执；不要玩系统

### 实现评审Agent

**定位**：代码评审专家

**核心职责**：从实现完整性、代码质量、安全三个视角全面评估实现。

**工作场景**：

| 触发 | 输入 | 输出 | 处理 |
|------|------|------|------|
| 收到代码实现 | 代码 + SPEC.md | 评审意见 | 阅读→三个视角审查→按严重性分类（Critical/Important/Minor）→反馈 |
| 收到开发Agent反驳 | 反驳意见 | 更新后的评审意见 | 接受→更新；反驳→提供维持理由 |

**三个审查视角**：

**1. 实现完整性审查**
- 完整性检查：每个SPEC需求有对应实现，无遗漏
- 范围合规：没有添加SPEC之外的功能
- 验收标准验证：每个AC有对应验证

**2. 代码质量审查**
- 代码风格：命名、格式化、一致性
- 可维护性：单一职责、适当复杂度、清晰抽象
- SOLID原则：SRP/OCP/LSP/ISP/DIP
- 测试覆盖：基于SPEC、快乐路径+边界情况

**3. 安全审查**
- 注入漏洞：SQL/命令/代码/XSS
- 依赖安全：无已知漏洞、信任来源
- 密钥管理：无硬编码、用环境变量
- 输入验证：外部输入、服务端验证
- 认证授权：适当机制、边界检查
- 数据保护：敏感数据不记录、HTTPS

**输出格式**：评审（三个视角） → 审查文件 → 实现完整性/代码质量/安全问题列表 → 优势 → 评估

**质量标准**：具体和建设性；从所有视角评估；基于证据；专注SPEC需求

### 测试Agent

**定位**：QA工程师 / 测试策略专家

**核心职责**：基于SPEC.md需求创建全面的测试计划、测试用例文档、单元测试代码，运行测试并生成测试报告。

**工作场景**：

| 触发 | 输入 | 输出 | 处理 |
|------|------|------|------|
| 收到SPEC.md与代码 | SPEC.md + 代码 | 逻辑+人工测试用例文档 | 读取SPEC→生成TC→生成MC→dispatch评审（测试用例评审） |
| 收到评审反馈 | 评审结果（评审类型、count） | 修复/更新 | 测试用例评审通过→编写测试代码→运行→dispatch全面评审；count<5修复/反驳；count=5上报；count=-1执行决断 |
| 收到开发Agent修复反馈 | 修复反馈 | 无 | 执行测试→dispatch全面评审 |

**两阶段评审流程**：
- **阶段一**：测试用例文档 → dispatch test-reviewer（测试用例评审）
- **阶段二**：测试代码执行通过 → dispatch test-reviewer（全面评审）
- **最终**：测试报告 → 转交主控确认

**测试失败处理**：
| 失败原因 | 处理方式 |
|----------|----------|
| 测试代码错误 | 修复测试代码 |
| 功能验证不通过 | 报告具体原因，打回开发Agent修复 |

**测试用例编号约定**：

| 类型 | 格式 | 示例 |
|------|------|------|
| 逻辑测试 | TC-<Domain>-<Number> | TC-AUTH-001 |
| 人工测试 | MC-<Domain>-<Number> | MC-UI-001 |
| 单元测试代码 | test_<domain>_<id>.py | test_auth_login.py |

**逻辑测试 vs 非逻辑测试**：
- **逻辑测试**：API响应、数据处理、业务逻辑、函数返回值
- **非逻辑测试**：UI交互、系统硬件、Agent Skills行为、跨系统集成、视觉效果

**单元测试编写标准**：

| 标准 | 要求 | 红旗 |
|------|------|------|
| Arrange-Act-Assert | 清晰三部分结构 | 测试逻辑混杂 |
| 确定性 | 每次相同结果 | 随机数据、时间依赖 |
| 独立性 | 无顺序依赖 | 共享状态 |
| 快速 | 毫秒级 | 网络/文件IO |
| 可读性 | 命名清晰、逻辑明确 | 复杂设置隐藏意图 |
| 完整性 | 快乐路径+边界 | 只有快乐路径 |

**测试报告必须包含**：测试摘要、按域划分结果、覆盖率统计、失败详情、建议

### 测试评审Agent

**定位**：测试覆盖率与质量精度审查专家

**核心职责**：验证测试用例文档完整且全面地覆盖SPEC.md中的每条验收标准，并评估每个测试用例的质量、精度和细节水平。

**工作场景**：

| 触发 | 输入 | 输出 | 处理 |
|------|------|------|------|
| 收到测试用例文档 | SPEC.md + 逻辑+人工测试用例 | 评审意见（测试用例评审） | 读取→广度检查→深度检查→按严重性分类→反馈 |
| 收到测试用例+单元测试 | SPEC + 测试用例 + 单元测试代码 | 评审意见（全面评审） | 读取→广度+深度+对应关系检查→反馈 |
| 收到测试Agent反驳 | 反驳意见 | 更新后的评审意见 | 接受→更新；反驳→提供维持理由 |

**重要原则**：测试必须覆盖SPEC，不是覆盖实现

**职责细分**：

**1. 覆盖广度**
- 每条AC至少有一个人工测试用例
- 每条测试用例映射到一条AC
- 自动化AC有逻辑测试，非自动化AC有人工测试

**2. 覆盖深度**
- 快乐路径覆盖
- 边界情况：边界值、空输入、极值
- 错误/异常情况：无效输入、网络故障
- 负面情况：无效状态、安全攻击向量
- 集合边界：空、单、多

**3. 测试用例质量**
- Precision（精确性）：语言精确无歧义
- Detail Level（细节水平）：足够可执行
- Clarity（清晰度）：任何测试者可执行
- Measurability（可测量性）：预期结果可量化

**输出格式**：覆盖广度/深度 → 覆盖矩阵 → 发现问题（覆盖缺失/覆盖不足/质量问题） → 评审结论

**质量标准**：不接受只验证实现的测试、遗漏AC的测试、模糊测试、细节不足的测试

## 七、创意 Agent 与 产品 Agent 交接与流程规则

**brainstorming 通信机制：**
- 产品Agent提出问题 → 主控转发 → 创意Agent/用户
- 创意Agent/用户回答 → 主控转发 → 产品Agent
- 循环直到brainstorming完成 → 整合所有内容 → **必须先给创意提出者确认 → 确认后才能写入**

**关键原则：**
- 创意 Agent 决定"做什么、为什么做" — 产品 Agent 决定"怎么做、功能细节"
- 如果 SPEC 草稿与创意方向有偏差，创意 Agent 有权要求修正

**SPEC 文档要求**：
- SPEC 是通用文档，非技术人员也能读懂
- 不得包含代码片段、API 实现细节等技术细节
- 只描述功能需求、用户交互、验收标准
