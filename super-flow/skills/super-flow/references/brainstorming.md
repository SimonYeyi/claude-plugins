# Brainstorming 需求澄清规范

---

## 对话模式

| 模式 | 适用场景 | 规则 |
|------|----------|------|
| **一次全问** | 创意模式 | 一次性针对Creative Brief提出多个有疑惑的问题 |
| **一次一问** | 产品模式 | 每次问一个问题，逐步确认，直到所有问题都已澄清 |

---

## 需求细节程度评估

**触发**：收到用户需求后，先评估细节程度，再决定挖掘策略

### 评估维度

| 维度 | 模糊信号 | 清晰信号 |
|------|----------|----------|
| **使用者** | "用户"、"大家" | 具体角色：运营、财务、admin |
| **场景** | "需要XX功能" | 在XX情况下，做XX操作，达到XX |
| **量化** | "快一点"、"方便" | 具体数字：<3秒、<5分钟 |
| **边界** | 未提及 | 明确排除什么、最多/最少 |
| **参照** | 无对比 | 提到现有方案/竞品/参考产品 |

### 细节程度分级

| 级别 | 特征 | 策略 |
|------|------|------|
| **L1 模糊** | <2个维度清晰 | 全维度挖掘，6个维度都要过 |
| **L2 粗略** | 2-3个维度清晰 | 按需挖掘，重点追问模糊维度 |
| **L3 具体** | 4-5个维度清晰 | 查漏补缺，确认边界 |
| **L4 完整** | 全部清晰 | 验证可行性，进入方案阶段 |

---

## 核心原则

**多选题优先、提出方案给建议、分部分确认、多轮迭代，直到完全澄清所有核心需求**

| 策略 | 说明 | 示例 |
|------|------|------|
| 一次一问 | 不同时提多个问题 | ✗ "你喜欢A还是B？C怎么样？还有一个选项D你想不想试试？" |
| 多选题优先 | 提供选项而非开放问题 | ✓ "A、B、C哪个更接近你的想法？" |
| 提出方案 | 不只问，还给建议 | ✓ "考虑到X，我建议A或B，你的优先级是？" |
| 分部分确认 | 逐步确认，不要最后一起确认 | ✓ "我们就这个流程达成一致了？" |
| 多轮迭代 | 需要多轮深入 | ✓ [第一轮确认范围 → 第二轮确认细节] |

---

## 提问格式规范

所有问题必须提供**一/二/三 序号** **A/B/C/D 选项**，明确列出每个选项的内容。**推荐选项**用`[推荐]`标注，并简要说明推荐理由。

最后一问在问题序号后面 提示"最后一个问题"。如：问题七（最后一个问题）：问题内容

```
问题一：问题内容

A. [选项A内容]
B. [选项B内容]
C. [选项C内容]
D. [选项D内容]

推荐 [B] — 理由：...
你的选择是？
```

---

## brainstorming对话加工

- 所有 brainstorming 对话必须记录（在上下文中交流，不写入文件）
- 最终**转化为结构化需求统计结果**：
  - 场景还原 → Overview（概述）+ User Stories（用户故事）
  - 根因分析 → Overview 的 Why 部分
  - 量化指标 → Acceptance Criteria（验收标准）
  - 约束条件 → Technical Constraints（技术约束）或单独章节
  - 成功标准 → Acceptance Criteria + Success Metrics（成功指标）

## 功能重叠分析（前置规范）

**目标**：明确新需求与项目现有功能的关系，避免重复建设

**触发条件**：探索项目现状后，判断用户需求与现有功能存在重叠

**追问模板**（根据重叠类型选择性追问）：
```
✓ “项目中已经有类似的功能吗？如果有，新需求和它的区别是什么？”
✓ “这个新功能是扩展现有功能、替代现有功能，还是完全独立的新功能？”
✓ “如果扩展现有功能，需要在哪些方面增强？哪些保持不变？”
✓ “如果替代现有功能，旧功能是否可以废弃？迁移成本如何？”
✓ “新功能是否需要与现有功能集成或交互？如何集成？”
```

**分类标准**：
| 类型 | 定义 | 处理策略 |
|------|------|----------|
| **扩展** | 在现有功能基础上增加能力 | 明确扩展点，保持向后兼容 |
| **替代** | 用新功能替换旧功能 | 制定迁移计划，评估废弃成本 |
| **独立** | 与现有功能并行存在 | 明确边界，避免功能冲突 |
| **互补** | 与现有功能配合使用 | 设计集成点，确保协同工作 |

**注意**：此规范在深度挖掘之前执行，用于确定需求定位

## 需求深度挖掘框架

### 维度1：使用者挖掘

**目标**：明确谁在用、谁受影响、谁决策

**追问话术**：
```
✓ "具体是谁来操作这个功能？"
✓ "还有其他角色会用到吗？"
✓ "谁最关心这个功能的实现效果？"
✓ "谁有权限操作这个功能？（管理员/普通用户/访客）"
```

**量化追问**：
```
✓ "大概多少人需要用到？"
✓ "每个用户每天/每周使用频率是多少？"
✓ "新手多还是老手多？"
```

---

### 维度2：场景挖掘

**目标**：还原真实使用场景，明确触发条件和完成标志

**追问话术**：
```
✓ "请描述一个具体的使用场景"
✓ "在什么情况下会用到这个功能？"
✓ "用户打开这个功能后，第一步做什么？最后一步做什么？"
✓ "这个流程从哪里开始，到哪里结束？"
```

**分支场景**：
```
✓ "如果出错了，用户会看到什么提示？"
✓ "如果网络不好会怎样？"
✓ "第一次用和以后用有什么区别？"
```

---

### 维度3：痛点挖掘

**目标**：量化现状问题，确定不做会怎样

**追问话术**：
```
✓ "现在是怎么处理这个问题的？"
✓ "现在的方式哪里让你不满意？具体是什么问题？"
✓ "这个问题出现频率是多少？每天/每周？"
✓ "这个问题大概影响多少人？"
```

**量化追问**：
```
✓ "每次处理这个问题大概花多少时间？"
✓ "每个月因为这个问题造成的损失（时间/金钱/人力）是多少？"
✓ "如果3个月不解决，问题会变严重吗？"
```

---

### 维度4：期望挖掘

**目标**：明确用户心中的"好"是什么

**追问话术**：
```
✓ "你希望这个功能上线后，具体改善什么？"
✓ "你见过类似的功能吗？喜欢哪一点？讨厌哪一点？"
✓ "如果只能保留一个核心功能，你会选哪个？"
✓ "有哪些功能你觉得是'必须有'，哪些是'最好有'？"
```

**边界追问**：
```
✓ "哪些情况是绝对不能出现的？"
✓ "有没有你觉得绝对不能砍掉的功能点？"
✓ "哪些操作你希望一步完成，而不是多步？"
```

---

### 维度5：量化挖掘

**目标**：将所有模糊描述转化为具体数字

**常见转化**：

| 模糊描述 | 必须追问 | 清晰定义示例 |
|----------|----------|--------------|
| "快速" | 具体时间？ | "首屏<2秒" |
| "方便" | 减少几步？ | "从5步减到2步" |
| "安全" | 什么标准？ | "符合等保三级" |
| "稳定" | 可用性指标？ | "99.9%可用" |
| "美观" | 参考什么？ | "参考飞书的设计语言" |
| "高效" | 提升多少？ | "减少50%操作时间" |
| "海量数据" | 具体量级？ | "单次导出<10万条" |
| "实时" | 延迟要求？ | "延迟<500ms" |

**追问话术**：
```
✓ "你说的'快'具体是多少秒？"
✓ "现在的数据量大概多少？增长预期呢？"
✓ "有没有可以参考的对标产品？它的指标是多少？"
✓ "你希望这个功能支持多少人同时使用？"
```

---

### 维度6：约束挖掘

**目标**：提前识别限制，避免后期返工

**技术约束**：
```
✓ "必须在哪些平台使用？（PC/手机/平板）"
✓ "需要兼容哪些浏览器或App版本？"
✓ "有没有必须对接的现有系统？"
✓ "有没有指定的技术栈？（框架/语言/数据库）"
```

**业务约束**：
```
✓ "有没有法规或合规要求？（GDPR/等保/行业标准）"
✓ "有没有特殊的权限或审批流程？"
✓ "上线时间有要求吗？有没有关键节点？"
```

**资源约束**：
```
✓ "有没有现成的设计稿或原型可以参考？"
✓ "有没有可以复用的组件或API？"
✓ "需不需要接入第三方服务？（支付/短信/地图）"
```

## 深度挖掘执行流程

### Step 1：评估细节程度

收到需求后，先对照[L1-L4分级表](#需求细节程度分级)评估当前需求的细节程度。

### Step 2：选择挖掘策略

| 级别 | 直接进入 | 需要挖掘的维度 |
|------|----------|----------------|
| L1 模糊 | 跳过 | 全部6个维度 |
| L2 粗略 | 使用者、场景 | 痛点、期望、量化、约束中模糊的 |
| L3 具体 | 使用者、场景、约束 | 痛点、期望、量化中模糊的 |
| L4 完整 | 验证可行性 | 按需补充 |

### Step 3：按序挖掘

根据评估结果，按优先级执行维度挖掘。每个维度用追问话术展开。

### Step 4：需求重述

用结构化方式复述理解：
```
"让我确认一下我的理解：
- 使用者：[谁]
- 使用场景：[什么情况下做什么]
- 痛点：[现在的问题，量化]
- 期望：[改善什么，有哪些必须有的功能]
- 量化指标：[速度/数量/频率等具体数字]
- 约束：[技术/业务/资源限制]

我理解得对吗？有什么遗漏？"
```

### Step 5：方案建议

基于澄清后的需求，给出1-3个方案选项，说明各自的优劣。

### Step 6：确认共识

用户对方案和优先级达成一致，进入下一阶段。

## 常见需求模式与应对策略

### 模式1：功能点清单
**特征**：用户列了一堆功能点，但没有优先级

**策略**：追问"如果只能做3个，选哪3个？"

### 模式2：对标竞品
**特征**：说"像XX产品那样"

**策略**：追问"具体喜欢XX产品的哪一点？讨厌哪一点？"

### 模式3：技术方案
**特征**：用户直接说"用XX技术实现"

**策略**：追问"为什么选这个技术？有什么约束吗？"

### 模式4：模糊愿景
**特征**："做一个平台"、"做生态"

**策略**：从"谁会用第一版"开始，逐步收窄范围

### 模式5：抱怨吐槽
**特征**：一直在说现在系统的问题

**策略**：引导到"如果解决了，你最想要什么？"

## 需求挖掘完成标准

### 必要条件（必须全部满足）

| # | 标准 | 检验问题 |
|---|------|----------|
| 1 | **使用者明确** | 能说出具体是谁在用、谁受影响、谁决策？ |
| 2 | **场景清晰** | 能描述一个完整的用户流程（起点→操作→终点）？ |
| 3 | **痛点量化** | 能说出当前问题造成的时间/金钱/人力损失？ |
| 4 | **期望具体** | 能说出1-3个"必须有"的功能点？ |
| 5 | **边界确认** | 能说出哪些是绝对不能砍掉的功能？ |

### 充分条件（满足任一即可）

| # | 标准 | 说明 |
|---|------|------|
| 1 | **用户确认** | 用户明确说"我理解的那样，开始做吧" |
| 2 | **无新增问题** | 连续2轮追问没有新的澄清需求 |
| 3 | **优先级排序** | 用户已完成核心功能优先级排序 |
| 4 | **方案选择** | 用户在提供的方案中做出了选择 |

### 快速通过条件

满足以下任一，可跳过剩余挖掘，直接进入方案阶段：
- 用户需求非常具体，提供了完整的功能列表和技术规格
- 用户明确说"先按这个做，有问题再调整"
- 现有功能已完全覆盖需求，只需做扩展

### 终止条件

满足以下任一，应停止挖掘，不应强行推进：
- 用户明确说"不知道，先做看"
- 用户拒绝回答任何追问
- 多次重述理解后用户仍有分歧

**原则**：挖掘是为了理解，不是为了完美。但**需求挖掘停止后没有继续挖掘的契机**，所以必须在澄清阶段做好充分的需求挖掘，达到必要条件和充分条件后才能停止挖掘。
